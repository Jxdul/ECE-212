F.3 Motivation
In assembly language programming, we need temporary data storage, because
• we have a limited number of registers
We may need to temporarily save and later restore the information contained
in the registers.
• we need to support subroutines and exception handlers
Therefore, reserve some memory for these purposes. To keep things as simple as
possible,
• this memory should be
• we need an easy way to keep track of how much of this temp. storage space is
used
For the latter requirement, all we need is one pointer to track either the location
of the last-used memory address or the first-available memory address. (More
detail on this later.)

Stack operations are FILO:
• When we push items onto the stack, we must pop the items off in reverse

) at the beginning and end of the subrou-

• modularity
• structured programming
• testability
• recoverability

Top-down design is a method to handle large, complex problems in a systematic
way:
• decomposing the problem into smaller, more tractable pieces
• separation of goals and means

F.6.3 Structured programming
Programs can be constructed from three fundamental components:
• sequences - linear set of steps
• loops — allow for a sequence to be repeated a number of times
• decision mechanisms — allow for one of two (or more) courses of action
F.6.4 Testability
Testability is a measure of how easy it is to test/ verify operation of a program or
component of a program; e.8.,
• accessibility of interfaces between modules
• breakpoints
• top-down vs. bottom-up testing
• white-box (internal operation) vs. black-box (functionality) testing

===== Page 75 =====
Some key points for PDL:
• shorthand notation for the description of algorithms using plain English to sub-
stitute for entire expressions
• compromise between high-level language and an assembly language program
• supports good programming principles (top-down design, modularity, struc-
tured programming)
• can be extended to deal with specific tasks
• try to use only English-like statements that precisely describe specific operations
• describe the meaning rather than how the approach will be implemented
• explicitly define control constructs (loops and decision mechanisms) - IF,
FOR, CASE, etc.
• design at a higher level than the target code (avoid syntactic elements from the
target assembly language)
• write at low enough level that coding from it will be nearly automatic
