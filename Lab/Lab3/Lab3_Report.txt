ECE 212 Lab 3 Report

Introduction

The objective of Lab 3 was to get more comfortable with ARM assembly subroutines, stack usage, and passing parameters between routines. In this lab we implemented three assembly routines: Welcomeprompt, Sort, and Display. Welcomeprompt collects the user input from MITTTY, Sort arranges the values in ascending order, and Display prints the final sorted results.

This lab also focused on proper stack discipline. Our routines had to preserve the return address and any registers that needed to survive subroutine calls, while also passing the number of entries and the array base address between routines without hardcoding assumptions. We used the provided C support functions such as printf, cr, value, and getstring to handle the MITTTY interaction.

Design (Including Flow Charts)

1. Overall program design

The program is split into three assembly files:
- Lab3a_L432KC.s implements Welcomeprompt.
- Lab3b_L432KC.s implements Sort.
- Lab3c_L432KC.s implements Display.

The test harness in Lab3_L432KC.s gives a menu that can test each subroutine on its own or run all three in order. The input values are stored starting at memory address 0x20001000, which matches the lab requirements. In the full program flow, Welcomeprompt is called first, Sort is called second, and Display is called last.

2. Welcomeprompt design

Welcomeprompt saves r4-r8 and lr at the start, then uses r4 as the current array pointer, r5 as the number of entries, r6 as the lower limit, r7 as the upper limit, and r8 as the input counter. The routine keeps asking for the number of entries until the user enters a value from 3 to 10. It then asks for the lower and upper bounds, and if the bounds are invalid it asks for them again. After that, it accepts the requested number of values one at a time, with a separate prompt for the final number. Each valid entry is stored in memory using post-indexed addressing, and any invalid entry causes an error message and another prompt.

At the end of the routine, the number of entries is written onto the stack so the later subroutines can use it. In our code this is done with `str r5, [sp, #24]` right before `POP {r4-r8, pc}`.

Welcomeprompt flowchart (from pre-lab/flowcharts/WelcomePrompt.md):

```mermaid
flowchart TD
    start([Start Welcomeprompt]) --> saveRegs[Save callee registers and keep baseAddress]
    saveRegs --> promptEntry[Prompt and read entryCount]
    promptEntry --> entryLow{"entryCount < 3?"}
    entryLow -->|Yes| errLow[Print entry-too-low error] --> promptEntry
    entryLow -->|No| entryHigh{"entryCount > 10?"}
    entryHigh -->|Yes| errHigh[Print entry-too-high error] --> promptEntry
    entryHigh -->|No| promptLower[Prompt and read lowerLimit]
    promptLower --> promptUpper[Prompt and read upperLimit]
    promptUpper --> boundsOk{"lowerLimit <= upperLimit?"}
    boundsOk -->|No| errBounds[Print bounds error and reprompt] --> promptLower
    boundsOk -->|Yes| initIndex[Set index = 0]

    initIndex --> doneInput{"index == entryCount?"}
    doneInput -->|Yes| writeStack[Write entryCount to stack flag]
    writeStack --> endNode([Restore registers and return])
    doneInput -->|No| isLast{"index == entryCount - 1?"}
    isLast -->|Yes| promptLast[Prompt for last number]
    isLast -->|No| promptValue[Prompt for number]
    promptLast --> readValue[Read numeric input]
    promptValue --> readValue
    readValue --> inRange{"lowerLimit <= value <= upperLimit?"}
    inRange -->|No| errRange[Print out-of-range error] --> isLast
    inRange -->|Yes| storeValue[Store value at memory and increment index]
    storeValue --> doneInput
```

3. Sort design

Sort receives the base address of the array in r0 and loads the entry count from the stack. The routine preserves r4-r7 and lr, then performs a bubble sort on the list in memory. The outer loop tracks how many passes are left, while the inner loop compares adjacent values and swaps them when the current value is greater than the next value. Since the routine uses the base address passed in r0, it does not depend on one fixed memory location.

Sort flowchart (from pre-lab/flowcharts/Sort.md):

```mermaid
flowchart TD
    start([Start Sort]) --> saveRegs[Save callee registers]
    saveRegs --> readInputs[Read entryCount from stack and arrayBase from R0]
    readInputs --> initOuter[Set i = 0]
    initOuter --> outerDone{"i >= entryCount - 1?"}
    outerDone -->|Yes| restoreEnd([Restore registers and return with stack unchanged])
    outerDone -->|No| initInner[Set j = 0]
    initInner --> innerDone{"j >= entryCount - 1 - i?"}
    innerDone -->|Yes| nextOuter[Increment i] --> outerDone
    innerDone -->|No| loadPair[Load array[j] and array[j+1]]
    loadPair --> compareVals{"array[j] > array[j+1]?"}
    compareVals -->|Yes| swapVals[Swap adjacent elements]
    compareVals -->|No| incInner[Increment j]
    swapVals --> incInner
    incInner --> innerDone
```

4. Display design

Display receives both the entry count and the array base address from the stack. After saving r4-r7 and lr, it prints the number of entries, prints a heading for the sorted output, then walks through the array and prints each value on its own line. When all values have been displayed, the routine prints "Program ended" and restores the stack before returning.

Display flowchart (from pre-lab/flowcharts/Display.md):

```mermaid
flowchart TD
    start([Start Display]) --> saveRegs[Save callee registers]
    saveRegs --> readInputs[Read entryCount and arrayBase from stack]
    readInputs --> printCountMsg[Print message for number of entries]
    printCountMsg --> printCountVal[Print entryCount value]
    printCountVal --> printSortedMsg[Print sorted-order message]
    printSortedMsg --> initIndex[Set index = 0]
    initIndex --> donePrint{"index >= entryCount?"}
    donePrint -->|Yes| printEnd[Print Program ended message]
    printEnd --> restoreEnd([Restore registers and return with stack unchanged])
    donePrint -->|No| loadVal[Load value at arrayBase + index*4]
    loadVal --> printVal[Print value]
    printVal --> nextIndex[Increment index]
    nextIndex --> donePrint
```

Testing

The provided test harness in Lab3_L432KC.s supports four menu options:
- Test 1 calls Welcomeprompt by itself and checks whether registers are restored and whether the number of entries is returned through the stack.
- Test 2 prints the values before sorting, calls Sort, and then prints the values after sorting so the bubble sort behavior can be observed directly.
- Test 3 calls Display and checks that registers and stack contents are restored correctly after the routine returns.
- Test 4 runs the complete sequence of Welcomeprompt, Sort, and Display.

The testing strategy checks both functionality and calling-convention correctness. This matters because the lab is not only about getting the right output, but also about preserving the stack pointer and protected registers across subroutine calls. The test harness explicitly prints "Register not properly restored" and "Stack not properly restored" if those requirements are violated.

Recommended functional test cases for this lab are:
- Entry count too low, such as 2, to verify the minimum-count error path.
- Entry count too high, such as 11, to verify the maximum-count error path.
- Lower and upper limits entered in the wrong order to verify the bounds error message.
- Input values below the lower limit or above the upper limit to verify range checking.
- Mixed unsorted values such as 6, 4, 45, 20, 10 to verify that bubble sort produces ascending output.
- Already sorted input to verify that Sort leaves correct input unchanged.
- Repeated values to verify stable handling of equal adjacent comparisons.

One thing worth noting is that our current Welcomeprompt code accepts the bounds only when `lowerLimit < upperLimit`, because it branches on `blt` after `cmp r6, r7`. The flowchart and assignment wording suggest the condition should be `lowerLimit <= upperLimit`. That means equal lower and upper limits would currently be rejected, so this should be treated as an edge case during testing.

Questions

Q1. Is it always necessary to implement either callee or caller preservation of registers when calling a subroutine? Why?

A1. No, it is not necessary to preserve every register every time a subroutine is called. What matters is that any register value that still needs to be used after the call must be preserved according to the calling convention. If the called routine may overwrite a register that the caller still needs, then that value must be saved either by the caller before the call or by the callee if that register is supposed to be preserved. In this lab, preservation matters because helper routines such as `printf`, `cr`, `value`, and `getstring` can overwrite registers, so we had to protect the values we still needed after those calls.

Q2. Is it always necessary to clean up the stack? Why?

A2. The stack must always be left in the expected state before control returns to the caller. If a subroutine pushes registers, allocates local storage, or adds temporary values to the stack, then those changes must be cleaned up properly. Otherwise the stack pointer will be wrong, return addresses may no longer line up correctly, and later subroutine calls can fail. The only exception is when the calling design intentionally leaves something on the stack for another routine to use, like passing a parameter or result. Even then, the stack still has to be restored correctly at the proper point in the program.

Q3. If a proper check for the getstring function was not provided and you have access to the buffer, how would you check to see if a valid number was entered?

A3. We could validate the number by scanning the input buffer one character at a time before converting it. First, we would check for an optional leading `+` or `-`. After that, every remaining character should be a digit from `0` to `9` until we reach the end of the string, a newline, or a null terminator. If the buffer is empty, contains only a sign, or contains any non-digit character such as a letter, space, or symbol in the middle of the number, then the input should be rejected as invalid. A full check could also include overflow detection while converting the digits into an integer. In simple terms, the idea is to make sure the buffer matches the pattern of a valid signed decimal number before accepting it.
